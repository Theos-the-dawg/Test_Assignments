|  **Q7.1 —
  Fault Tolerance (3 marks)**  |  Fault
  Tolerance is the ability of a system to continue operating correctly even
  when one or more of its components fail. (1 mark) A
  bank's online system might achieve this through: data replication across
  multiple servers so a server failure doesn't cause data loss (1 mark);
  automatic failover where a backup server takes over if the primary fails; and
  load balancers that redirect traffic away from failed nodes (1 mark for any

| valid mechanism). |  |
| ----------------- | - |

|  **Q7.2 —
  Client-Server vs Microservices (3
  marks)**  |  Client-Server
  Architecture: A central server provides services to multiple client machines.
  Advantage: Centralized data management and security — all data is managed in
  one place, making backups and access control simpler. (1 mark comparison +
  0.5 advantage)   Microservices
  Architecture: The system is broken into small, independent services that each
  handle a specific function and communicate via APIs. Advantage: Each service
  can be deployed, updated, and scaled independently without affecting the rest

| of the system. (1 mark comparison + 0.5 advantage) |  |
| -------------------------------------------------- | - |

|  **Q8 — System
  Documentation (6 marks — 1.5 per
  documentation type)**  |  Product
  Documentation: Audience = developers & stakeholders. Poor consequence =
  system built incorrectly, misaligned with requirements.   Process
  Documentation: Audience = project managers & development team. Poor
  consequence = missed deadlines, no audit trail, project derailment.   User
  Documentation: Audience = end users. Poor consequence = high error rates,
  increased support calls, user frustration.   Technical
  Documentation: Audience = developers & system admins. Poor consequence =
  maintenance becomes extremely difficult; new developers cannot understand or
  extend the codebase.   (Award

| 0.5 audience + 1 consequence per type.) |  |
